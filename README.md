# KEPSA CircularMap — DuckDB + FastAPI + Frontend Demo

A GIS MSME mapping platform demo for the KEPSA Tujenge Pamoja consultancy:
map view, dashboard, and full CRUD data manager, backed by a real DuckDB
database through a FastAPI REST API.

## What's here

```
kepsa-gis/
├── backend/
│   ├── app/
│   │   ├── main.py          FastAPI app, CORS, router registration
│   │   ├── database.py      DuckDB connection + write lock
│   │   ├── models.py        Pydantic request/response models
│   │   └── routers/
│   │       ├── msmes.py     CRUD + filters + dashboard stats
│   │       └── export.py    GeoJSON / CSV / Parquet export
│   ├── data/
│   │   ├── msme_data.csv    Source seed data (820 synthetic MSMEs)
│   │   └── kepsa.duckdb     The actual database file (generated by seed_db.py)
│   ├── seed_db.py           Rebuilds kepsa.duckdb from the CSV
│   └── requirements.txt
└── frontend/
    └── index.html           Single-page UI (Leaflet map, Chart.js dashboard,
                              CRUD table) — calls the backend over fetch()
```

## Two adaptations from the original architecture, and why

1. **Flat/denormalized schema, not normalized counties/sub_counties/wards
   tables.** For 800 rows and a single analyst, three extra joins on every
   query buys referential-integrity guarantees you don't need yet. The
   `msmes` table keeps county/sub_county/ward as plain text columns.
   Migrating to the normalized version later (if KEPSA needs strict
   integrity or multi-source data entry) is a straightforward follow-on —
   the API contract wouldn't need to change, only the queries behind it.

2. **Vanilla JS frontend, not React/Vite.** The UI (map, dashboard, CRUD
   table) already existed as a working single-page app; rewiring its data
   layer to call this REST API was faster and lower-risk than a parallel
   React rebuild, and produces the same interaction pattern. If you want
   the React/Vite version specifically (e.g. because your team will
   maintain this long-term and wants component structure), say so and I'll
   build it — the API underneath doesn't change either way.

Also skipped: DuckDB's `spatial` extension (`ST_Point`, R-tree index) —
it needs to download from `extensions.duckdb.org`, which wasn't reachable
from the sandbox this was built in. Latitude/longitude columns do the same
job for this dataset size and are what ArcGIS/QGIS/Google Earth expect as
X/Y fields anyway. If your own machine can reach that domain, `LOAD spatial;`
and swapping `latitude, longitude` for `ST_Point(longitude, latitude) AS geom`
in `seed_db.py` gets you real geometry types and an R-tree index with no
other changes needed.

## Running it locally

**Backend:**
```bash
cd backend
pip install -r requirements.txt
python3 seed_db.py            # builds data/kepsa.duckdb from the CSV
uvicorn app.main:app --reload --port 8000
```
Visit `http://localhost:8000/api/health` to confirm it's up, or
`http://localhost:8000/docs` for interactive Swagger docs of every endpoint.

**Frontend:**
Just open `frontend/index.html` directly in a browser — it defaults to
`http://localhost:8000` as the API base. To point it elsewhere (e.g. a
deployed backend), set `window.KEPSA_API_BASE` before the main script runs,
either by editing the constant in `index.html` or injecting it via a small
`<script>` tag above it.

## API reference (also live at `/docs` once running)

| Method | Path | Purpose |
|---|---|---|
| GET | `/api/msmes` | List with `county`, `value_chain`, `ownership`, `search`, `sort_col`, `sort_dir`, `page`, `limit` |
| GET | `/api/msmes/stats` | Dashboard aggregates (same filters) |
| GET | `/api/msmes/{id}` | Single record |
| POST | `/api/msmes` | Create |
| PUT | `/api/msmes/{id}` | Update |
| DELETE | `/api/msmes/{id}` | Delete |
| GET | `/api/export/geojson` | Full dataset as GeoJSON (ArcGIS/QGIS/Google Earth) |
| GET | `/api/export/csv` | Full dataset as CSV (Power BI, Excel) |
| GET | `/api/export/parquet` | Full dataset as Parquet |

All of these were tested end-to-end (create → read → update → delete →
verify 404, plus all three export formats) before this was handed over.

## Deploying it for real (I can't do this step from here)

I don't have accounts on Render/Railway/Fly.io/Vercel, so I can't actually
click deploy for you — but the app needs zero code changes to go live:

**Backend (Render, free tier is fine for this dataset size):**
1. Push `backend/` to a GitHub repo.
2. New Web Service on Render → connect the repo.
3. Build command: `pip install -r requirements.txt && python3 seed_db.py`
4. Start command: `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
5. Note: Render's free tier disks are ephemeral on redeploy — the
   `python3 seed_db.py` in the build step re-seeds every deploy, which is
   fine here since the CSV is the source of truth. If you start editing
   data in production and want it to persist across redeploys, add a
   Render persistent disk mounted at `backend/data/`.

**Frontend (Vercel/Netlify, or just GitHub Pages since it's one static file):**
1. Push `frontend/` to a repo (or the same repo, different root).
2. Deploy as a static site.
3. Set `window.KEPSA_API_BASE` in `index.html` to your Render backend URL
   before pushing.
4. Update `allow_origins` in `backend/app/main.py` from `["*"]` to your
   actual frontend domain once you know it — the wildcard is fine for a
   demo, not for anything real.

## What this is standing in for in the application

This whole package is a working sample for Requirement #6 ("samples of
GIS dashboards or related work") and gives evaluators something to run
and click through rather than just read about. The `.duckdb` file itself
also doubles as tangible evidence for Section A of the scope of work — a
real, inspectable geospatial database, not just a claim in a proposal.
